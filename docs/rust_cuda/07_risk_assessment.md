# Rust-CUDA Migration: Risk Assessment & Fallback Strategy

Status: pre-Wave-A planning artefact for the 0.3 milestone. Reviewed before the toolchain spike; revisited at every wave gate.

The migration replaces ~18 hand-emitted PTX kernels in `src/jit/*.rs` with Rust source compiled through `rustc_codegen_nvvm`. The toolchain was un-archived in late 2024; we are placing a bet that it stays alive. This document enumerates what could go wrong, how we would notice, and how we get out.

---

## 1. Risk inventory

| ID | Risk | Prob. | Impact | Detection | Mitigation |
|----|------|-------|--------|-----------|------------|
| R1 | rust-cuda toolchain re-archived / abandoned again | medium | critical | Weekly check of repo commit cadence + GitHub watch on Rust-CUDA org | Pin a known-good commit by `rev =` in Cargo.toml; keep hand-emit code in `git` history reachable by tag `v0.2-handemit`; pre-commit Fallback B plan. |
| R2 | `rustc_codegen_nvvm` breaks under a future Rust nightly | high | high | CI matrix bumps the pinned nightly weekly; codegen panics or PTX diff | Pin nightly via per-member `rust-toolchain.toml` in the `kernels/` workspace crate; host crate stays on stable; bump only with a green bench run. |
| R3 | rust-cuda PTX triggers driver bugs hand-emit avoids (e.g. odd `cuModuleLoadDataEx` shapes) | low | high | `CUDA_ERROR_*` on load or first launch; differential test against hand-emit | Keep a "golden-answer" differential harness in `tests/` that runs both backends for one release; ptxas `--verbose` diff at CI time. |
| R4 | Machine-generated PTX is 10–30% slower than hand-tuned, eating 0.2 perf wins | medium | high | Criterion bench delta vs `v0.2` baseline per kernel | Per-kernel rollback flag (see Fallback A); keep `#[inline(always)]` and shared-memory layout annotations close to hand-emit; cherry-pick hot kernels back to hand-emit if needed. |
| R5 | Loss of dynamic kernel parameterisation. Today `compile_partition_reduce_kernel_multi(n_vals)` emits one PTX per N at runtime; rust-cuda is compile-time monomorphic | high | medium | Compile-time error or combinatorial PTX bloat | Two-tier strategy: variants-per-N for small N (1..=8) generated by macro; const-generic kernels behind a feature flag for research; for unbounded N keep the hand-emit path indefinitely. |
| R6 | Build-time explosion (30 s – 5 min added to `cargo build`) | high | medium | `cargo build --timings` on CI clean target | Compile kernels in a separate workspace member with its own target dir; cache PTX artefacts in CI; gate the kernel rebuild behind a feature so doc-only builds skip it. |
| R7 | CI complexity: `cuda-stub` for docs.rs vs real CUDA on bench host, now plus rust-cuda toolchain | medium | medium | CI matrix grows; flaky stub builds | Three-row matrix: (stub, stable), (real-cuda, stable, hand-emit), (real-cuda, nightly, rust-cuda). Make rust-cuda gated by `--features rust-cuda` so stub path is untouched. |
| R8 | Contributor friction — new contributors need rust-cuda set up | high | low | Drop-off on PRs touching kernels; questions in issues | `kernels/README.md` with one-command install; document the cuda-stub path as the default for non-kernel work; CONTRIBUTING.md gates kernel changes behind a clear "you need the toolchain" callout. |
| R9 | CUDA driver version mismatch — rust-cuda's NVVM targets specific CUDA versions; bench host driver may be older | low | high | NVVM emits PTX with unsupported sm_/`.target` lines; load failure | Detect at install (`nvcc --version` + `cuModuleLoadDataEx` smoke test); document minimum driver in `kernels/README.md`; if bench host can't upgrade, stay on hand-emit for that environment. |
| R10 | Supply chain — rust-cuda adds cust, cust_core, cust_raw, cuda_std, etc. | low | medium | `cargo audit` warnings; `cargo deny` policy violations | Add `cargo audit` + `cargo deny` to CI before Wave A; pin minor versions in Cargo.toml; review transitive crate list once at start of Wave A and again at Wave G. |

Highest-leverage risks: **R1 (critical impact)**, **R2 (high prob × high impact)**, **R5 (structural, blocks Tier-2 multi-key kernels)**.

## 2. Fallback ladder

Three escalating fallbacks; each is reversible up to the next one.

### Fallback A — keep hand-emit + rust-cuda in parallel

- **Mechanism**: per-kernel feature flags (`kernel-partition-handemit`, `kernel-shmem-sum-handemit`, ...). Default features pick rust-cuda; user / CI can flip.
- **Cost**: ~doubled maintenance for one release cycle. We pay for both code paths during 0.3 and prune the loser in 0.4.
- **Trigger**: any single bench regression > 5% AND post-mortem points at PTX shape (not at host-side launch config, scheduling, or memory).
- **Rollback**: flip the per-kernel flag for the offending kernel; ship 0.3 mixed.

### Fallback B — revert the migration entirely

- **Mechanism**: `git revert` the `kernels/` workspace member; restore `src/jit/*_kernel*.rs` from the `v0.2-handemit` tag (which we cut at Wave A's start).
- **Cost**: 0.3 slips ~2 months; all rust-cuda work goes into a `rust-cuda-experiment` branch and is shelved.
- **Trigger**: toolchain abandoned mid-wave (R1 fires); OR > 3 kernels regress > 10% and Fallback A would mean rust-cuda owns < half the kernels (the migration's not paying for itself).
- **Rollback**: a clean revert + a CHANGELOG note. The branch survives for a future attempt.

### Fallback C — pivot to `cust_raw` (FFI without NVVM)

- **Mechanism**: keep PTX hand-emit, but use `cust_raw` for the host-side CUDA driver calls instead of our current direct FFI through `cudarc`. Drops the NVVM codegen story but keeps a stable Rust surface to the driver.
- **Cost**: ~1 week to spike; 0.3 redefined as a host-API modernisation instead of a kernel-language change. Less dramatic but more achievable.
- **Trigger**: NVVM-backend has rotted but cust_raw still works (R1 / R2 partial). Or Wave A go/no-go fails on PTX correctness (R3) but the host-side API surface is sound.
- **Rollback**: cust_raw is a thinner dep set; rolling it back to direct FFI is mechanical.

## 3. Decision triggers — Wave A go/no-go checklist

Evaluate at the end of Wave A (the toolchain spike, ~2 weeks). All five must be checked to proceed to Wave B.

- [ ] `partition_kernel` compiles via `rustc_codegen_nvvm` against the pinned Rust nightly.
- [ ] Output PTX loads via `CudaModule::from_ptx` and produces bit-identical results to hand-emit on the canonical test fixture.
- [ ] Bench delta on the spike kernel ≤ 5% vs hand-emit on the bench host (3 runs, median).
- [ ] Build time impact on full `cargo build` (clean target, all features): ≤ +60 s.
- [ ] Successfully pinned Rust nightly via per-workspace-member `rust-toolchain.toml`; the host crate `javelin` still builds on stable.

If any box stays unchecked → stop. Choose between Fallback C (if the failure is NVVM-specific) or deferring the milestone to 0.4 (if the failure is structural — e.g. const-generics not viable for R5). Do not push into Wave B with red boxes; the cost of unwinding doubles every wave.

## 4. Recovery timeline if R1 hits mid-migration

Scenario: we are at the end of Wave D (Tier-1 GROUP BY kernels migrated, ~halfway). The Rust-CUDA org has had no commits for 3 months.

**Day 0–7 — monitor.** Don't react. First commits often resume quietly after a maintainer break; archived-then-revived projects sometimes show this pattern. Keep building and benching against the pinned commit. No public statement.

**Day 7–30 — entrench.** Snapshot the rust-cuda commit we depend on; pin in Cargo.toml as `rev = "..."` (not branch). Mirror the repos under a `javelin-org/` GitHub mirror as cold backup. Assume from here on that we are self-maintaining the toolchain. Write up the "if you must patch rust-cuda" doc in `docs/rust_cuda/`.

**Day 30+ — decide.** Three branches:

- **(a) Fork upstream**: take ownership of `Rust-CUDA/Rust-CUDA` as `javelin-org/rust-cuda`, accept the maintenance burden. Justified only if we expect to ship multiple kernels per release. Probably not worth it for a single product.
- **(b) Finish migration with the snapshot.** Accept that no upstream updates come. Works if the snapshot is stable against our pinned nightly. Re-evaluate at 0.4.
- **(c) Fallback B** (revert the migration). The right call if Wave E-onward kernels need toolchain features we haven't yet exercised, because we'd be debugging the toolchain ourselves without upstream help.

The choice at Day 30 hinges on: (i) is the snapshot complete enough for the remaining waves? (ii) how active is the broader rustc_codegen ecosystem (other backends, cg_clif) — a healthy ecosystem signals that NVVM-style backends remain viable.

## 5. Instrumentation — what we watch week-by-week

Concrete metrics, tracked in a wave-status spreadsheet:

- **Bench delta per kernel** vs `v0.2-handemit` baseline. Specifically: Tier-1 q1 / q4 and Tier-2 q5 / q3, plus the partition-reduce micro-bench. Threshold: > 5% regression triggers Fallback A discussion; > 10% on 3+ kernels triggers Fallback B.
- **`cargo build` wall time** on a clean target, full features. Threshold: +60 s at Wave A end, +120 s by Wave G end. Beyond that, R6 has materialised.
- **rust-cuda repo health**: commits/week, open-issue close rate, last release tag. If commits/week drops to 0 for 3 consecutive weeks, raise R1 status from "watching" to "active".
- **Lines of PTX-emitting Rust deleted**. Non-economic but motivating; an honest measure of whether the migration is actually reducing maintenance surface area or just relocating it.
- **PTX size delta per kernel**. Bigger PTX isn't always slower, but a 5× blow-up suggests const-generic monomorphisation is over-expanding (R5).
- **New transitive deps** introduced into `cargo tree`. Audit at Wave A end and Wave G end.

---

## Net assessment

The migration is **worth attempting** but **not worth defending past Wave A**. The upside — deleting ~18 PTX-emitting Rust files, gaining type safety in kernels, opening the door to const-generic parameterisation of group-by tier kernels — is real and durable. The downside is bounded by Fallback B: we can revert cleanly because the kernels' boundary is well-defined and the hand-emit code lives in git history under a tag. The single risk that justifies a hard stop is R1 fired before Wave A is green; once we are past the go/no-go gate with R1 dormant and R2 / R4 / R5 measured at green, the residual risk is mostly performance tuning rather than structural failure. We go in with a pre-cut tag, a pre-written Fallback B PR, and a willingness to back out at the Wave A gate.
